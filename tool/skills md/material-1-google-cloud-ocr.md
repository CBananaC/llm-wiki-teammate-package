## Layout Notes

Material 1 uses a vertical Traditional Chinese book layout.

- Reader order is right-to-left by vertical column, then top-to-bottom within each column.
- Finder/PDF page numbers are not the same as printed page numbers. Always record both when possible.
- Example already checked: Finder page 43 corresponds to printed page 38.
- The printed page number appears near the lower outside corner.
- Some pages include a vertical right-side header with collection/book section information and document title.
- Some pages include small marginal notes such as 「硃批」.
- OCR output must preserve the source wording. Do not silently modernize, translate, simplify, or normalize historical wording.
- OCR text is raw evidence only after visual checking. Mark it as OCRed, not verified, until manually checked against page images.

## Google Cloud OCR Method

Use Google Cloud Document AI OCR, not local OCR, when the user asks for Google Cloud OCR.

Project metadata is recorded in [[google-cloud-info]]. Do not store credentials, access tokens, refresh tokens, service-account JSON, or private keys in the wiki.

Required runtime conditions:

- `gcloud` is authenticated, or `GOOGLE_OAUTH_ACCESS_TOKEN` is temporarily set in the shell.
- `pdftoppm` and `pdfseparate` are available.
- A Document AI OCR processor exists and its processor ID is known.
- Network access to `oauth2.googleapis.com` and `documentai.googleapis.com` works.

If network access fails, stop and report the failure. Do not substitute local OCR unless the user explicitly approves a fallback, and if a fallback is used, mark it clearly as non-Google OCR.

Known processor behavior:

- Processor `a6087517993e9c3a` returned `Unsupported mime type for content layout parser: image/png` when sent PNG images.
- For this processor, send each page as a single-page PDF with MIME type `application/pdf`.

## Discover Processor ID

List processors without saving credentials:

```bash
ACCESS_TOKEN="$(gcloud auth print-access-token)"
curl -sS \
  -H "Authorization: Bearer ${ACCESS_TOKEN}" \
  "https://documentai.googleapis.com/v1/projects/delta-entry-496910-e7/locations/us/processors"
```

Use the OCR processor ID from the returned processor list. If processors are in another location, replace `us` with that location.

## Cleanup Guideline For 《明清臺灣檔案彙編》

Use this cleanup guideline after raw OCR has been generated for Material 1.

### Document Boundaries

- A line beginning with `《` and ending with `》` marks the source title for a new document.
- The next non-empty line after the source title is the document title.
- The next non-empty line after the document title usually contains position/author, document date, and document type.
- Text after that metadata line and before the next `《...》` source marker is the document content.
- A document may continue across Finder/PDF pages. Preserve all Finder page numbers covered by that document.

### Metadata Parsing

For the metadata line:

- Extract the final parenthesized value as document type, e.g. `（上奏）` or `（其他）`.
- Extract the Chinese reign-date before the document type as the document date.
- Treat the remaining preceding text as `職銜＋作者`.
- If no official position is present, keep the author name in `職銜＋作者`.

### Body Cleanup

- Remove Arabic printed page numbers inside the OCR text, such as `38`, `39`, `40`, `41`, `42`.
- Keep Finder/PDF page numbers as metadata in `Finder頁碼`.
- Remove OCR line-break spaces inside Chinese text.
- Convert half-width punctuation to full-width Chinese punctuation:
  - `,` → `，`
  - `:` → `：`
  - `;` → `；`
  - `(` and `)` → `（` and `）`
  - `?` and `!` → `？` and `！`
- Do not silently correct suspected OCR errors. Leave them in place until manually checked against page images.
- Combine all body paragraphs into one `正文：` line.
- Convert full Chinese reign dates by following [[tool/skills md/chinese-reign-date-conversion]]. Preserve the original Chinese date and append the Gregorian date in full-width parentheses.

### Final Date And Imperial Reply

- In the final few lines of a document, a short line containing a Chinese reign-date with `年`, `月`, and `日` should be recorded as `文末日期`.
- Lines beginning with `【珠批】` or `【硃批】` are imperial red-ink replies. Normalize the label to `【硃批】` and record the line under `硃批`.
- If a `【硃批】` appears inline at the end of a content line, split it from the preceding content and record it as `硃批`.

### Clean TXT Output Format

Each document should use this format:

```text
第N件
來源：《...》
題名：...
職銜＋作者：...
文首日期：...
文書類型：...
Finder頁碼：...

正文：...

文末日期：...
硃批：...
```

Leave `文末日期：` or `硃批：` blank if no value is present.

When converting this cleaned TXT into JSON, use `文首日期`, not `日期`.

### JSON Field Format

Some single text fields contain smaller bibliographic or prosopographic relationships. In JSON, encode those relationships directly as compact arrays.

For `來源`, use:

```json
"來源": ["宮中檔乾隆朝奏摺", "第六十二冊"]
```

The order is `[系列／檔名, 冊次]`.

For `職銜作者`, use:

```json
"職銜作者": ["福建陸路提督", "任承恩"]
```

The order is `[職銜, 作者]`.

Only split fields when the relationship is clear. If one part is absent, use an empty string for that position, e.g. `["", "林爽文"]`.
